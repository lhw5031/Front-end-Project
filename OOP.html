객체지향 프로그래밍

Java OOP 문서 Python Classes

OOP는 프로그램을 데이터와 기능을 가진 객체들의 협력으로 바라보는 방식입니다. 코드를 역할과 책임 단위로 나누기 때문에 규모가 커져도 구조를 이해하고 변경하기 쉬워집니다.

Java

class User {
  String name;

  void login() {
    System.out.println(name + " logged in");
  }
}

Python

class User:
    def __init__(self, name):
        self.name = name

    def login(self):
        print(f"{self.name} logged in")

객체(Object)

객체는 상태와 행동을 함께 가진 독립적인 단위입니다. 현실의 사용자, 주문, 게시글처럼 프로그램 안에서 의미 있는 대상을 표현합니다.

Java

User user = new User();
user.name = "Jin";
user.login();

Python

user = User("Jin")
user.login()

클래스(Class)

클래스는 객체를 만들기 위한 설계도입니다. 어떤 속성을 가지고 어떤 메서드를 실행할 수 있는지 정의합니다.

Java

class Product {
  String name;
  int price;
}

Python

class Product:
    def __init__(self, name, price):
        self.name = name
        self.price = price

인스턴스(Instance)

인스턴스는 클래스로부터 실제로 생성된 객체입니다. 같은 클래스로 만든 인스턴스라도 각자 다른 값을 가질 수 있습니다.

Java

Product keyboard = new Product();
Product mouse = new Product();

Python

keyboard = Product("Keyboard", 50000)
mouse = Product("Mouse", 20000)

속성(Attribute)

속성은 객체가 가진 데이터입니다. Java에서는 보통 필드라고 부르고, Python에서는 인스턴스 변수라고 부르는 경우가 많습니다.

Java

class User {
  String name;
  int level;
}

Python

class User:
    def __init__(self, name, level):
        self.name = name
        self.level = level

메서드(Method)

메서드는 객체가 수행할 수 있는 행동입니다. 객체의 속성을 읽거나 변경하고, 다른 객체와 협력하는 코드를 담습니다.

Java

void levelUp() {
  level++;
}

Python

def level_up(self):
    self.level += 1

캡슐화

캡슐화는 데이터와 동작을 하나로 묶고, 객체 내부 상태를 외부에서 함부로 바꾸지 못하게 보호하는 방식입니다.

Java

class Account {
  private int balance;

  void deposit(int amount) {
    if (amount > 0) balance += amount;
  }
}

Python

class Account:
    def __init__(self):
        self._balance = 0

    def deposit(self, amount):
        if amount > 0:
            self._balance += amount

상속

상속은 부모 클래스의 기능을 자식 클래스가 물려받는 기능입니다. 공통 코드를 재사용하고, 더 구체적인 클래스를 만들 때 사용합니다.

Java

class Member extends User {
  int point;
}

Python

class Member(User):
    def __init__(self, name):
        super().__init__(name)
        self.point = 0

다형성

다형성은 같은 요청을 보내도 실제 객체에 따라 다른 동작이 실행되는 특징입니다. 코드가 구체 클래스보다 공통 타입에 의존하게 만듭니다.

Java

Payment payment = new CardPayment();
payment.pay(10000);

Python

payment = CardPayment()
payment.pay(10000)

추상화

추상화는 중요한 특징만 드러내고 복잡한 구현은 숨기는 방식입니다. 사용자는 내부 구조보다 제공되는 기능에 집중할 수 있습니다.

Java

interface Payment {
  void pay(int amount);
}

Python

class Payment:
    def pay(self, amount):
        raise NotImplementedError

생성자

생성자는 객체가 만들어질 때 초기 상태를 설정하는 특별한 메서드입니다. 필수 값이 빠지지 않도록 객체 생성 시점에 받아둘 수 있습니다.

Java

class User {
  String name;

  User(String name) {
    this.name = name;
  }
}

Python

class User:
    def __init__(self, name):
        self.name = name

this

this는 현재 객체 자신을 가리킵니다. Python에서는 같은 역할을 관례적으로 self라는 이름으로 표현합니다.

Java

this.name = name;
this.level = level;

Python

self.name = name
self.level = level

접근 제어자

접근 제어자는 클래스 내부의 데이터와 기능을 어디까지 공개할지 정합니다. Java는 키워드로 명확히 제어하고, Python은 이름 규칙을 주로 사용합니다.

Java

public class User {
  private String password;
  protected int level;
}

Python

class User:
    def __init__(self):
        self._level = 1
        self.__password = "secret"

Getter / Setter

Getter와 Setter는 객체의 값을 읽고 변경하는 통로입니다. 값을 검증하거나 내부 표현을 숨기고 싶을 때 유용합니다.

Java

public String getName() {
  return name;
}

public void setName(String name) {
  this.name = name;
}

Python

@property
def name(self):
    return self._name

@name.setter
def name(self, value):
    self._name = value

오버로딩

오버로딩은 같은 이름의 메서드를 매개변수에 따라 다르게 정의하는 것입니다. Java는 문법으로 지원하고, Python은 기본값이나 가변 인자로 비슷하게 표현합니다.

Java

int add(int a, int b) {
  return a + b;
}

int add(int a, int b, int c) {
  return a + b + c;
}

Python

def add(*numbers):
    return sum(numbers)

오버라이딩

오버라이딩은 부모 클래스의 메서드를 자식 클래스에서 다시 정의하는 것입니다. 다형성을 가능하게 만드는 대표적인 기법입니다.

Java

class CardPayment extends Payment {
  @Override
  void pay(int amount) {
    System.out.println("card");
  }
}

Python

class CardPayment(Payment):
    def pay(self, amount):
        print("card")

추상 클래스

추상 클래스는 공통 구현을 일부 제공하면서, 자식 클래스가 반드시 구현해야 할 메서드를 정할 때 사용합니다.

Java

abstract class Report {
  abstract String title();

  void print() {
    System.out.println(title());
  }
}

Python

from abc import ABC, abstractmethod

class Report(ABC):
    @abstractmethod
    def title(self):
        pass

인터페이스

인터페이스는 객체가 반드시 제공해야 하는 메서드의 약속입니다. 구현보다 역할을 먼저 정의할 때 사용합니다.

Java

interface Notifier {
  void send(String message);
}

Python

from typing import Protocol

class Notifier(Protocol):
    def send(self, message: str) -> None:
        ...

연관 관계

연관 관계는 한 객체가 다른 객체를 알고 사용하는 관계입니다. 두 객체의 생명주기가 강하게 묶이지는 않습니다.

Java

class OrderService {
  void create(User user) {
    System.out.println(user.name);
  }
}

Python

class OrderService:
    def create(self, user):
        print(user.name)

집합 관계

집합 관계는 전체가 부분을 가지고 있지만, 부분 객체가 전체와 독립적으로 존재할 수 있는 관계입니다.

Java

class Team {
  List<User> members;

  Team(List<User> members) {
    this.members = members;
  }
}

Python

class Team:
    def __init__(self, members):
        self.members = members

합성 관계

합성 관계는 전체 객체가 부분 객체를 직접 만들고 소유하는 강한 관계입니다. 전체가 사라지면 부분도 함께 사라지는 구조로 이해할 수 있습니다.

Java

class Page {
  private Header header;

  Page() {
    this.header = new Header();
  }
}

Python

class Page:
    def __init__(self):
        self.header = Header()

SOLID 원칙

SOLID는 객체지향 설계를 유지보수하기 쉽게 만드는 다섯 가지 원칙입니다. 역할 분리, 확장성, 대체 가능성, 작은 인터페이스, 의존성 역전을 강조합니다.

SRP

클래스는 하나의 책임에 집중합니다.

OCP

확장에는 열려 있고 수정에는 닫혀 있어야 합니다.

LSP

자식 클래스는 부모 타입을 대체할 수 있어야 합니다.

ISP / DIP

작은 인터페이스와 추상화에 의존하는 설계를 지향합니다.

Java

class OrderService {
  private Notifier notifier;

  OrderService(Notifier notifier) {
    this.notifier = notifier;
  }
}

Python

class OrderService:
    def __init__(self, notifier):
        self.notifier = notifier

객체지향의 장점

객체지향은 복잡한 코드를 역할 단위로 나누고, 재사용 가능한 구조를 만들기 좋습니다. 변경이 필요한 부분을 특정 객체나 클래스에 모을 수 있어 유지보수에 유리합니다.

재사용성

상속, 조합, 인터페이스로 이미 만든 코드를 다시 활용할 수 있습니다.

유지보수성

책임이 분리되면 변경 영향 범위를 파악하기 쉬워집니다.

확장성

새 기능을 별도 클래스로 추가하기 좋은 구조를 만들 수 있습니다.

모델링

현실의 개념을 코드 구조로 옮기기 쉽습니다.

객체지향의 단점

객체지향은 잘못 설계하면 클래스가 너무 많아지고 관계가 복잡해질 수 있습니다. 작은 문제에 과한 추상화를 적용하면 오히려 이해하기 어려운 코드가 됩니다.

설계 비용

역할과 책임을 잘 나누기 위해 초기에 고민이 필요합니다.

복잡도 증가

상속 계층이 깊어지면 흐름을 추적하기 어려워질 수 있습니다.

과한 추상화

필요 이상으로 인터페이스와 클래스를 만들면 생산성이 떨어집니다.

성능 부담

상황에 따라 객체 생성과 동적 dispatch가 비용이 될 수 있습니다.
